# 08 · Usabilidad · Internacionalización y localización

> Soporte técnico de múltiples idiomas y culturas: externalización de strings,
> formato de datos con locale, RTL, pluralización y encoding. La i18n mal
> implementada se descubre cuando el primer usuario no hispanohablante intenta usar el producto.
>
> **Marcos de referencia:** Unicode CLDR · ICU Message Format · WCAG 2.2 (idioma declarado) · GDPR (idioma de consentimiento).

---

## A. Externalización de strings

#### `UX-I18N-001` — Strings de UI externalizados, sin literales hardcodeados
**Severidad:** high · **Tags:** `i18n` · **Aplica a:** frontend · backend

Todo texto visible por el usuario proviene de archivos de traducción o un
sistema de i18n, no de literales embebidos en el código. Un string hardcodeado
es texto que nunca se podrá traducir sin modificar el código.

**Dónde buscar:** `**/*.{tsx,jsx,vue,svelte,html}`, `**/*.{ts,js,py,rb,go,java,cs}`, `**/components/**`, `**/views/**`, `**/templates/**`
**Patrones:**
- `>\s*[A-ZÁÉÍÓÚÑ][a-záéíóúñ\s,\.!?]{10,}\s*<`                               # texto literal en JSX/HTML (>10 chars)
- `["']\s*(Bienvenido|Welcome|Error|Guardar|Cancel|Submit|Loading)[a-z\s]*["'](?!\s*:)` # strings UI típicos hardcodeados
- `console\.log\b`                                                              # excluir logs (no son UI)
- `t\(['"]\w[\w.]+['"]\)|i18n\.t\(|__\(|_\(|trans\(`                         # patrones correctos esperados
- `\$t\(['"]\w|useTranslation|useI18n|gettext`                                 # frameworks i18n esperados
**Señal de N/A:** herramienta interna de uso exclusivo por un equipo monolingüe, sin requisito de i18n documentado.

**Verificar:**
- [ ] Los textos visibles al usuario usan funciones de traducción (`t()`, `i18n.t()`, `__()`, etc.).
- [ ] Los archivos de traducción existen y están completos para todos los idiomas soportados.
- [ ] Las claves de traducción son descriptivas (`errors.required_field`) no genéricas (`error1`).
- [ ] Hay un proceso para detectar strings sin traducir (lint, CI check).

**Banderas rojas:**
- `<button>Guardar cambios</button>` sin función de traducción.
- `alert("El campo es requerido")` hardcodeado.
- Mezcla de strings traducidos y literales en el mismo componente.

#### `UX-I18N-005` — Mensajes sin concatenación de strings traducidos
**Severidad:** high · **Tags:** `i18n` · **Aplica a:** frontend · backend

Los mensajes que incluyen variables (nombre de usuario, cantidades, fechas) usan
interpolación del sistema i18n, no concatenación. La concatenación rompe en
idiomas con orden de palabras diferente al español/inglés.

**Dónde buscar:** `**/*.{ts,js,tsx,jsx,vue,svelte,py,rb,go,java,cs}`, `**/i18n/**`, `**/locales/**`, `**/translations/**`
**Patrones:**
- `t\(['"][^'"]+['"]\)\s*\+\s*\w+|\w+\s*\+\s*t\(['"][^'"]+['"]\)`            # concatenación con resultado de t()
- `["']\w[\w\s]*["']\s*\+\s*(user\.name|userName|count|total|date)`           # concatenación de string + variable
- `\`[^`]*\$\{t\([^)]+\)\}[^`]*\$\{[^`]+\}\``                                # template literal mezclando t() y vars
- `t\(['"][^'"]+\{\{[^}]+\}\}['"]\)|t\(['"][^'"]+\{[^}]+\}['"]\)`            # patrón correcto con interpolación
**Señal de N/A:** sistema sin mensajes dinámicos con variables (solo labels fijos).

**Verificar:**
- [ ] Los mensajes con variables usan interpolación: `t('greeting', { name: user.name })`.
- [ ] Los archivos de traducción contienen el placeholder: `"greeting": "Hola, {{name}}"`.
- [ ] No hay concatenación del tipo `t('hello') + ' ' + user.name + '!'`.
- [ ] Los mensajes con plurales usan la API de pluralización, no `count > 1 ? t('items') : t('item')`.

**Banderas rojas:**
- `"Hola, " + t('user') + "!"` — el orden de palabras varía por idioma.
- `t('you_have') + count + t('messages')` — rompe en árabe, ruso, etc.

## B. Formato de datos con locale

#### `UX-I18N-002` — Fechas, números y monedas formateados con locale
**Severidad:** medium · **Tags:** `i18n` `l10n` · **Aplica a:** frontend · backend

El formato de fechas, números y monedas varía por cultura. `1,000.50` (en-US)
es `1.000,50` (es-ES). Mostrar el formato incorrecto genera confusión y errores.

**Dónde buscar:** `**/*.{ts,js,tsx,jsx,vue,svelte,py,rb,go,java,cs}`, `**/utils/**`, `**/helpers/**`, `**/formatters/**`
**Patrones:**
- `\.toFixed\(\d+\)|Math\.round\([^)]+\)\s*/\s*100`                           # número formateado sin locale
- `new Date\([^)]+\)\.toString\(\)|\.toLocaleDateString\(\)`                  # fecha sin locale explícito
- `\$\s*[\d,\.]+|\d+\s*(USD|EUR|CLP|MXN)`                                      # moneda formateada a mano
- `Intl\.NumberFormat|Intl\.DateTimeFormat|Intl\.RelativeTimeFormat`           # patrón correcto esperado
- `toLocaleString\(\s*locale|formatDate\(|formatCurrency\(`                    # helpers esperados
**Señal de N/A:** producto mono-locale con documentación explícita de que solo opera en una cultura/región.

**Verificar:**
- [ ] Las fechas usan `Intl.DateTimeFormat` con locale explícito o una librería (date-fns, Luxon) configurada con locale.
- [ ] Los números usan `Intl.NumberFormat` con el locale del usuario.
- [ ] Las monedas usan `Intl.NumberFormat` con `style: 'currency'` y `currency: 'CLP'` (o el código correspondiente).
- [ ] El locale usado es el del usuario, no un hardcodeado `'es-ES'` para todos.

**Banderas rojas:**
- `price.toFixed(2)` para mostrar precios en la UI sin formato de moneda.
- `new Date(ts).toString()` para mostrar fechas a usuarios.
- `"$" + amount` para formatear moneda manualmente.

#### `UX-I18N-004` — Pluralización correcta por locale (no concatenación)
**Severidad:** medium · **Tags:** `i18n` · **Aplica a:** frontend · backend

Distintos idiomas tienen reglas de pluralización distintas. El inglés tiene 2
formas (1 item / 2 items). El árabe tiene 6. El ruso tiene 3. La lógica
`count === 1 ? singular : plural` solo funciona en inglés.

**Dónde buscar:** `**/*.{ts,js,tsx,jsx,vue,svelte,py,rb}`, `**/i18n/**`, `**/locales/**`
**Patrones:**
- `count\s*===\s*1\s*\?\s*t\(|count\s*==\s*1\s*\?\s*['"]`                     # ternario manual de pluralización
- `count\s*>\s*1\s*\?\s*t\(|n\s*!==\s*1\s*\?\s*t\(`                           # variantes del ternario manual
- `_plural|one:|other:|few:|many:|zero:`                                        # claves de plural en archivos de traducción
- `t\(['"][^'"]+['"],\s*\{\s*count:|i18nKey.*count|plural\s*\(`                # patrón correcto con count
**Señal de N/A:** sistema sin mensajes que varíen según cantidad.

**Verificar:**
- [ ] Los mensajes pluralizables usan la API de plural de la librería i18n (`t('items', { count })` con claves `one/other`).
- [ ] Los archivos de traducción tienen todas las formas requeridas por el idioma target.
- [ ] No hay lógica `if count === 1` embebida en el código de UI para seleccionar strings.

**Banderas rojas:**
- `count === 1 ? 'mensaje' : 'mensajes'` en el código del componente.
- Archivos de traducción con solo `"items": "items"` sin variante singular.

## C. Soporte técnico de escritura

#### `UX-I18N-003` — Soporte RTL evaluado y probado
**Severidad:** medium · **Tags:** `i18n` `l10n` `rtl` · **Aplica a:** frontend

El árabe y el hebreo se escriben de derecha a izquierda. Un layout LTR roto
en RTL es inutilizable para esos usuarios. La detección y el soporte es una
decisión técnica que debe tomarse antes de lanzar en esos mercados.

**Dónde buscar:** `**/*.{css,scss,sass,less}`, `**/*.{tsx,jsx,vue,svelte,html}`, `**/tailwind.config.*`, `**/styles/**`
**Patrones:**
- `dir\s*=\s*['"]rtl['"]|direction:\s*rtl|html\[dir=rtl\]`                    # soporte RTL presente
- `margin-left|padding-left|text-align:\s*left|border-left`                    # propiedades LTR no lógicas
- `margin-inline-start|padding-inline|border-inline`                           # propiedades lógicas CSS (correcto)
- `rtl:|[dir=rtl]\s|logical\s`                                                 # Tailwind RTL o CSS lógico
**Señal de N/A:** producto explícitamente fuera de mercados RTL (ej: solo LATAM/Europa, documentado).

**Verificar:**
- [ ] Si se soportan idiomas RTL: el `<html>` o el contenedor raíz tiene `dir="rtl"` cuando corresponde.
- [ ] Los estilos usan propiedades CSS lógicas (`margin-inline-start` en lugar de `margin-left`) o hay overrides RTL.
- [ ] Los íconos direccionales (flechas, chevrons) se invierten en RTL.
- [ ] Hay tests o screenshots de la UI en modo RTL para los componentes principales.

**Banderas rojas:**
- Producto lanzando en árabe/hebreo con layout LTR sin ningún soporte RTL.
- `float: left` y `text-align: left` hardcodeados en todos los componentes sin alternativa RTL.

## D. Encoding y fundamentos

#### `UX-I18N-006` — Encoding UTF-8 en toda la pila (DB, API, UI)
**Severidad:** high · **Tags:** `i18n` `encoding` · **Aplica a:** backend · data · frontend

El encoding incorrecto produce caracteres corruptos (`Ã©` en vez de `é`,
`?????` en vez de caracteres asiáticos). UTF-8 debe ser el encoding en cada
capa: base de datos, conexión de DB, API responses, HTML y archivos de texto.

**Dónde buscar:** `**/database.{yml,yaml,json,ts,js,py}`, `**/db.config.*`, `**/connection*.{ts,js,py}`, `**/nginx*.conf`, `**/apache*.conf`, `**/*.html`, `**/*.{sql,prisma}`
**Patrones:**
- `charset\s*=\s*latin1|charset\s*=\s*iso-8859|encoding.*latin`                # encoding no-UTF8 en DB
- `charset\s*=\s*utf8(?!mb4)|collation.*utf8(?!mb4)`                          # MySQL utf8 (no utf8mb4 — incompleto para emojis)
- `charset=utf-8|charset=UTF-8|<meta.*charset.*utf-8`                         # patrón correcto esperado
- `Content-Type.*charset=utf-8|'charset':\s*'utf-8'`                          # headers correctos esperados
- `CONVERT\s*\(|CHARSET\s*\(`                                                  # conversiones forzadas (señal de problema)
**Señal de N/A:** sistema sin texto multilingüe ni caracteres fuera de ASCII (ej: sistema numérico puro).

**Verificar:**
- [ ] La base de datos usa charset `utf8mb4` (MySQL/MariaDB) o `UTF8` (PostgreSQL) con collation adecuada.
- [ ] La cadena de conexión a la DB especifica el encoding: `charset=utf8mb4`, `client_encoding=UTF8`.
- [ ] Las respuestas HTTP tienen `Content-Type: application/json; charset=utf-8`.
- [ ] Los archivos HTML tienen `<meta charset="UTF-8">` como primer elemento del `<head>`.
- [ ] Los archivos de traducción están guardados en UTF-8 sin BOM.

**Banderas rojas:**
- `collation: utf8_general_ci` en MySQL (trunca emojis y algunos caracteres CJK).
- Respuestas API sin `charset` en el `Content-Type`.
- Archivos `.properties` o `.po` guardados en Latin-1.
